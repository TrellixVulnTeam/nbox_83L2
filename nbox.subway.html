Subways#

These function are meant to make jobs easier and contain the code for what I am calling Subway, which converts any OpenAPI spec into an RPC-like interface. At the end of the day, each API call is really nothing but a functional call with added steps like serialisation and networking combined. This kind of added complexity can:

  1. Make code look really ugly, think try-catches and r.raise_for_request()

  2. Easy to develop because information has to be packaged as different parts for each type of function call (GET, POST, …). The true intent was always to pass it the relevant information and forget about underlying details.

  3. API endpoints are just strings and it’s easy to have a typo in the URL. When the ``Subway``is loaded with the OpenAPI spec, it will disallow incorrect URLs. while managing the parameters correctly.

  4. A more pythonic way of programming where you can use the dot notation.

Based os these ideas there are three types of subways:

  1. Subway: Does not load OpenAPI spec and will blindly call the API, avoid this

  2. Sub30: Built for OpenAPI v3.0.0, this becomes nbox_ws_v1

  3. SpecSubway: This is used with the FastAPI’s OpenAPI spec, this is used in systems that use FastAPI (eg. Compute Server)

Like Subway but built for Nimblebox Webserver APIs.

Usage:

ws = Sub30(
  _url = "https://my-web.site/",
  _session = nbox_session,
  _api = loads(fetch("https://my-web.site/openapi.json", True).decode()),
)
Parameters

  • _url (str) – the base url of the API

  • _api (dict) – OpenAPI json dict

  • _session (requests.Session) – Session object to use for requests

  • prefix (str, optional) – This is internal, do not use it explicitly.
